# Implementation Plan for Secrets Feature with Key Rotation

## Overview
Implement a secrets management system similar to variables, with encryption at rest and workspace-level encryption keys. Secrets will be stored in workspace schemas (not public schema) and made available to functions via a `SECRETS` global variable. Includes support for encryption key rotation.

## Encryption Layer

### 5. Encryption Utilities
- **File**: `crates/service_utils/src/encryption.rs` (new file)
  - Add encryption/decryption functions using AES-256-GCM
  - Functions:
    - `generate_encryption_key()` - Generate random 32-byte key, return as base64
    - `encrypt_secret(plaintext: &str, key: &str) -> Result<String>`
    - `decrypt_secret(ciphertext: &str, key: &str) -> Result<String>`
    - `decrypt_with_fallback(ciphertext: &str, current_key: &str, previous_key: Option<&str>) -> Result<String>`
      - Try current key first, fall back to previous key if decryption fails
  - Keys stored in workspace table, potentially encrypted with AWS KMS in production

    - Return `KeyRotationStatus` (success/failure, number of secrets re-encrypted)
  - All read operations decrypt using `decrypt_with_fallback()` to handle key rotation gracefully

### 8. Key Rotation Logic
- **File**: `crates/context_aware_config/src/api/secrets/key_rotation.rs` (new file)
  - `rotate_workspace_key(conn, workspace_id, org_id, user, change_reason) -> Result<KeyRotationStatus>`
    1. Fetch workspace to get current encryption_key
    2. Fetch all secrets for workspace (with current encrypted values)
    3. Generate new encryption key
    4. Begin database transaction
    5. For each secret:
       - Decrypt with current key using `decrypt_secret()`
       - Re-encrypt with new key
       - Update encrypted_value and increment key_version
    6. Update workspace: new key → encryption_key, old key → previous_encryption_key, set key_rotation_at
    7. Commit transaction
    8. Log rotation event in audit log
    9. Return `KeyRotationStatus` with count and timestamp
  - Error handling for partial rotation failures (rollback transaction)
  - Validate user has permission for key rotation

## Security Considerations

### 13. Key Rotation Best Practices
- Rotation should be done during low-traffic periods (UI warning)
- Previous key retained for grace period (configurable, default 7 days)
- Rotation logged in audit trail with timestamp and user
- Consider implementing automatic rotation schedule (future enhancement)
- Rate limiting on rotation endpoint (prevent abuse)
- Require special permission/role for key rotation (workspace admin)
- Workspace encryption keys encrypted with KMS in PROD/SANDBOX

## Key Differences from Variables
1. **Smithy Model**: Separate `secret.smithy` with masked/unmasked response types
2. **Storage Location**: Secrets table in workspace schemas (not public)
3. **Encryption**: Values encrypted at rest using workspace-specific keys
4. **Key Versioning**: Track which encryption key version encrypted each secret
5. **Key Rotation**: Full workspace key rotation with automatic re-encryption via dedicated endpoint
6. **UI Masking**: Values shown once at creation with copy button, always masked thereafter
7. **Global Variable**: Available as `SECRETS` (vs `VARS` for variables)
8. **Security**: Workspace encryption keys encrypted with KMS in production
9. **Audit Trail**: Key rotations logged with timestamp, user, and count of re-encrypted secrets
10. **API Response**: Two response types - masked for normal ops, unmasked only for create
